Mortgage Loan Officer & Borrower Copilot — MVP

A mortgage copilot that helps loan officers (LOs) and processors pre‑qualify W‑2 borrowers by combining a conversational borrower intake, automated document understanding, and multi‑agent reasoning — always with strict human‑in‑the‑loop control before anything is shown to the borrower or persisted as final.

1. System overview

A web application with a React (Next.js App Router) front end, a FastAPI backend, and a LangGraph + AWS Bedrock agent layer that calls AWS Textract for OCR/IDP. State is persisted in a relational database via SQLAlchemy 2.0 + Alembic; documents live in object storage; every agent run is traced in LangSmith.

The Phase‑1 MVP delivers the end‑to‑end loop for a single W‑2 borrower: borrower chat intake → document upload → Textract‑backed classification and extraction → eligibility & conditions draft → LO review and approval of extractions, status, and borrower‑facing messages. The borrower never sees pre‑qualification status or conditions until a loan officer has explicitly approved them.

Operating principle. Every generative output produced by the agent layer is treated as a draft. Drafts are persisted alongside human corrections and approval flags; only approved artifacts cross the boundary back to the borrower.
Structured LLM outputs (2026‑05‑29). Agent completions that carry business data are constrained to Pydantic schemas via get_structured_model() (ChatBedrockConverse.with_structured_output). Borrower intake uses a tiered pipeline (L0 regex → L1 schema-bound extraction → L2 templated or structured reply) so most chat turns incur zero Bedrock tokens while ambiguous natural language still routes to the LLM. Database writes always pass through LoanApplicationData validation in try_apply_application_patch.

2. Context, goals & non‑goals

Phase‑1 goals

  • Borrower can submit a simplified application and upload key documents via a conversational web UI.
  • Automatically classify and OCR pay stubs, W‑2s, bank statements, and a synthetic 1003‑like PDF.
  • Extract structured income, asset, and applicant metadata into a normalized DealContext.
  • Produce a draft pre‑qualification (DTI/LTV + green/yellow/red) and conditions list for LO review.
  • Generate editable internal rationale and borrower‑facing explanations — always behind an approval gate.
  • Demonstrate a multi‑agent architecture (intake, document understanding, eligibility, explanation, orchestrator) with full LangSmith traces.

Phase‑1 non‑goals

  • No real credit pulls, pricing engines, or binding credit decisions.
  • No investor‑specific guideline matrices (Conventional vs FHA/VA/USDA).
  • No support for self‑employed income, non‑traditional income, or multi‑borrower files.
  • No production LOS integration; Phase‑1 stops at internal "reviewed pre‑qual notes."
  • No mobile native apps; responsive browser only.
  • No RAG over real policy documents (extensibility hook for a later phase).

3. Logical architecture

Five logical tiers cooperate to support the borrower → LO loop: client apps, the FastAPI backend, an agent layer built on LangGraph, persistent state in a relational DB and object storage, and a set of external managed services (Bedrock, Textract, LangSmith).

Frontend Backend service Orchestrator Agent Data store External / managed

Component domains

Frontend Borrower Portal

Chat-style intake, inline document upload prompts, live status panel showing captured fields and remaining gaps. Scoped to a single deal via a signed borrower session token.

Frontend LO / Processor Console

Deal list with status filters, application summary, side-by-side document viewer + extraction editor, eligibility panel, and the message approval workspace that enforces the human-in-the-loop gate.

Backend FastAPI APIs

Thin routers per domain (auth, deals, borrower_chat, documents, extractions, eligibility, messages) with business logic isolated in services/. Access control is enforced by require_deal_access.

Orchestrator LangGraph state machine

Single build_graph() entry point that wires the four agent nodes, holds the typed LoanCopilotState, and routes per-trigger (borrower turn, document upload, run-eligibility).

Agents Four cooperating LLM agents

Intake, Document Understanding, Eligibility & Conditions, Explanation & Messaging. Shared I/O contracts live in schemas/agent_io.py; Bedrock calls use get_structured_model(schema). Intake is tiered (L0/L1/L2) for token efficiency. None bypass the approval gate.

Data Persistence

SQLAlchemy 2.0 with a portable JSON column strategy (SQLite locally, JSONB on Postgres). Documents in an object-storage abstraction with local-filesystem and S3 backends.

External Bedrock & Textract

Bedrock Converse via langchain-aws: get_chat_model() for free-form use and get_structured_model(PydanticSchema) for schema-bound JSON (no prompt-side json.loads). Model ID is env-configurable (e.g. Claude 3 Haiku locally, Sonnet in prod). Textract analyze_document / analyze_expense as before.

External LangSmith observability

One project per agent (borrower_chat, doc_understanding, eligibility, messaging). Run metadata includes deal_id, document_id, and user_id for correlation with backend logs.

5. Data model overview

State flows from a Borrower through a single Deal, accumulating a LoanApplication (from chat), a DealContext (from documents and human corrections), Documents with their Extractions, and the dual internal/borrower Messages draft. ChatTurns retain conversation history and ApprovalEvents form an immutable audit of human gates.

State flow through the entities

  1. Borrower → Deal. One borrower in Phase‑1 owns one deal; status starts as intake_in_progress.
  2. ChatTurn → LoanApplication. Each borrower turn invokes the Intake Agent: L0 intake_slots extracts a patch, L1 LoanApplicationPatch (structured Bedrock) runs only when L0 is not confident, then try_apply_application_patch deep-merges and validates against LoanApplicationData before commit. L2 replies use templates when progress was made; otherwise IntakeTurnResult. When missing_fields is empty, the orchestrator transitions Deal.status to docs_pending.
  3. Document → DocumentExtraction → DealContext. Each upload creates a Document; the background pipeline produces a DocumentExtraction whose normalized values are merged additively into DealContext. Human corrections always win over machine values.
  4. DealContext + LoanApplication → Eligibility. The Eligibility Agent reads both, computes DTI/LTV deterministically via rules.py, then asks the LLM to explain and refine the conditions list — never to change the status, DTI, or LTV.
  5. Eligibility → Messages. The Explanation & Messaging Agent writes internal_draft and borrower_draft. Both stay invisible to the borrower until borrower_approved=True.
  6. ApprovalEvent. Every gate crossing — extraction edits, eligibility approval, per-channel message approval — appends an immutable audit row.

6. Agent orchestration view

Four agents and one orchestrator form a LangGraph state machine over a typed LoanCopilotState. Each agent owns a single concern, calls only the tools granted to it, and emits structured outputs that are persisted by the backend rather than printed straight to the user.

Production LLM schema pattern

All agents that persist machine-generated business data share one pattern: define a Pydantic agent I/O schema, call Bedrock through services/bedrock/client.get_structured_model(schema) (LangChain with_structured_output), then validate and merge results in application code before writing to the database. This replaces ad-hoc “return JSON only” prompts and json.loads repair paths.

ModuleRole
schemas/agent_io.py Canonical structured-output contracts: LoanApplicationPatch, IntakeTurnResult, MessageDrafts, ConditionRefinement, DocumentClassification.
services/bedrock/client.py get_chat_model() and get_structured_model(schema, tags, temperature) — single factory for Bedrock + schema binding.
agents/intake_slots.py L0 deterministic slot parsing and is_l0_confident() gate for L1 escalation.
agents/tools/application_writer.py get_application_snapshot, apply_application_patch, try_apply_application_patch (rollback on ValidationError).
schemas/loan_application.py Authoritative LoanApplicationData — every patch is merged then validated here before LoanApplication.data is committed.

Tiered borrower intake (L0 → L1 → L2)

Each POST /borrower/chat/messages turn runs through three tiers to balance language understanding with token cost:

TierMechanismTypical tokensWhen it runs
L0 extract_patch_from_message in intake_slots.py (regex / heuristics) 0 Every turn; extracts all slots the message clearly states.
L1 get_structured_model(LoanApplicationPatch) Small (schema-bound) Only when L0 did not confidently answer the next missing field (e.g. ambiguous amounts).
L2 Template _template_reply or get_structured_model(IntakeTurnResult) 0 or small Reply only: templates when a patch applied or L1 was skipped; LLM reply when extraction failed entirely.
Structured output vs tool calling. The provider forces the completion into the given Pydantic shape (structured output). The intake agent does not expose application_writer as an LLM-invoked tool; the backend calls try_apply_application_patch after extraction. Tool calling remains available for future nodes but is not used for intake persistence in the current implementation.

Other agents — structured schemas

  • Eligibility: rules.py computes status, DTI, LTV, and condition templates; ConditionRefinement may only update title and rationale per condition id.
  • Messaging: MessageDrafts with deterministic template fallback when Bedrock is unavailable.
  • Document understanding: filename/OCR heuristics first; DocumentClassification when confidence < 0.75; extraction_mapper remains pure functions over OCR output.

7. End-to-end sequence flows

The MVP supports one happy path with five clearly separated stages. Each stage is independently retryable and each crosses a documented persistence boundary before the next stage can run.

Flow narrative

  1. Borrower applies via chat. The borrower receives a signed session token scoped to their deal, then converses with the Intake Agent. Each turn runs L0 regex extraction, optionally L1 LoanApplicationPatch structured output, then try_apply_application_patch to validate and persist LoanApplicationData. Replies use templates when possible. When missing_fields is empty, the orchestrator transitions Deal.status to docs_pending.
  2. Documents are classified and extracted. Each upload writes raw bytes to object storage and enqueues run_document_pipeline. Classification uses filename/OCR heuristics first, then DocumentClassification structured output when confidence is low; Textract OCR and extraction_mapper produce normalized JSON merged into DealContext.
  3. LO / processor reviews extractions. The console renders normalized fields beside the document viewer; PUT /documents/{id}/extraction stores corrections in human_corrections and re-merges DealContext. Corrections always win.
  4. Eligibility & conditions are computed. POST /deals/{id}/eligibility/run runs rules.py for DTI/LTV/status and condition templates, then optional ConditionRefinement structured output (titles/rationales only). The Messaging Agent produces MessageDrafts via structured output with template fallback.
  5. Messages are generated and approved. The LO edits and approves the eligibility result, the conditions list, and per-channel (internal / borrower) messages. Only after borrower_approved=True does the borrower's next GET /borrower/application include the approved message — surfaced inline as a system turn in ChatWindow.

8. Non-functional & extensibility

Security

  • All endpoints authenticated: staff via JWT, borrower via a signed session token scoped to a single deal_id.
  • require_deal_access dependency enforces the scoping invariant on every deal_id-bearing endpoint; verified by integration tests and re-verified end-to-end by Playwright in Phase 5.
  • All data in transit over HTTPS; data at rest encrypted (database + object storage).
  • Synthetic-only data in Phase‑1: fixtures and seeds use placeholder PII tokens.
  • Approval invariant enforced server-side by approval_service.py: borrower endpoints never expose internal_draft, and never expose borrower_draft until borrower_approved=True. The UI is a secondary defense.

Observability

  • LangSmith runs tag Bedrock calls by purpose: intake_extract, intake_reply, document_understanding, eligibility, messaging (plus legacy project names per agent). Run metadata includes deal_id, document_id, user_id.
  • Structured JSON logs via structlog; X-Request-Id correlates HTTP requests with LangSmith run_id values.
  • EventLog table (Phase 6) records intake_complete, document_extracted, eligibility_computed, and messages_approved with timestamps.
  • Health probes: /healthz (liveness) and /readyz (database + storage + Bedrock model reachable).

Performance

  • Target: single document classify + OCR + map within 5–10 seconds in the MVP environment.
  • Document pipeline runs asynchronously via FastAPI BackgroundTasks; the API surface remains compatible with a future swap to RQ or Celery for durable retries.
  • Centralized exponential-backoff retries inside the Bedrock and Textract service wrappers (Phase 6).
  • In-process token bucket rate-limits on POST /borrower/chat/messages and POST /documents.
  • Bedrock token efficiency: tiered intake (L0/L1/L2) avoids an LLM call on most chat turns; document classification uses the LLM only when heuristic confidence < 0.75.

Deployability

  • Single Docker Compose stack in dev (postgres + localstack + backend + frontend) with the same env-var contract as deployed environments.
  • Backend and frontend ship as independent container images (Gunicorn/Uvicorn and Next.js standalone respectively).
  • Database portability (SQLite ↔ Postgres) is preserved by sticking to SQLAlchemy-portable types and the built-in JSON column.
  • Terraform IaC and GitHub Actions are explicit future work (Phase 6+).

Extensibility hooks

More document types

Add a new predicted_type enum value, a new extraction_mapper.map_* function, and a few golden tests. The Document Understanding Agent picks it up via its structured-output classifier.

RAG over policy docs

Plug a vector store behind a new tool exposed to the Eligibility & Messaging agents. State and orchestration are unchanged; only the prompt context grows.

LOS integration

The current approval_service already represents the integration boundary. A future LOS adapter consumes approved Messages, EligibilityResult, and LoanApplication via existing schemas.

Real credit pulls / pricing

Add a new agent node behind a feature flag; the orchestrator can sequence it between Eligibility and Messaging without touching existing nodes.

Multi-borrower deals

Promote the Borrower ↔ Deal relationship to many-to-many through a join table; LoanApplicationData already nests identity, contact, employment, income, and assets per borrower in its schema shape.

Durable background work

Swap FastAPI BackgroundTasks for RQ or Celery behind the existing workers/document_pipeline.run_document_pipeline entry point — no API surface change required.

Known risks (acknowledged, mitigated, or deferred)

  • OCR accuracy variability on synthetic documents. Mitigated by mandatory LO/processor review (Phase 5 ExtractionReview) where human corrections always override machine values.
  • Regulatory positioning of "pre-qualification" vs "pre-approval" is explicitly out of scope; borrower-facing surfaces carry a "Draft — not yet approved by LO" banner.
  • Guideline depth beyond DTI/LTV thresholds is deferred to later phases that will likely require RAG over real policy documents and updated data models / prompts.
  • AWS regional parity between Bedrock and Textract must be confirmed before deploy; services/ocr/base.py abstracts the OCR provider so we can swap if entitlement is delayed.
